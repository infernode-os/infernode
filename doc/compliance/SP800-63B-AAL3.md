## 1. What AAL3 requires (and what InferNode does)

AAL3 (SP 800-63B §4.3 / §5) requires, in essence:

| AAL3 requirement | InferNode mechanism | Status |
|------------------|---------------------|--------|
| **Hardware-based authenticator** | YubiKey (FIDO2), secret never leaves the key | ✅ |
| **Verifier-impersonation (phishing) resistance** | FIDO2 credential bound to relying-party / channel; challenge-response, not a typed shared secret | ✅ |
| **Authenticator + verifier; possession proven each auth** | Login requires the physical key present each session (touch) | ✅ |
| **User verification (something-you-know/are) on the authenticator** | **FIDO2 PIN (UV)** — hardware-verified on the key | ✅ shipped, hardware-verified |
| **No exportable authentication secret** | `hmac-secret` output `R` derived on-key; only `R` (touch/UV-gated) ever crosses the boundary | ✅ |
| Authenticator lifecycle (recovery, multiple authenticators) | Recovery slot ✅; **dual-key default ☐**, **DK save-back ☐** | partial |

Assurance-level mapping as shipped (`doc/yubikey-2fa-operations.md` §intro):

| Mode | Login requires | 800-63B |
|------|----------------|---------|
| Touch-only | secstore password + key **presence** | AAL2 |
| **UV (PIN)** | secstore password + key + **hardware-verified FIDO PIN** | **AAL3** |

## 2. The load-bearing AAL3 property — and its proof

The distinguishing AAL3 control is **hardware user-verification**: presence (touch) alone
is AAL2; AAL3 additionally requires the authenticator to verify the user. InferNode uses
the FIDO2 PIN, verified *on the key*, and the PIN is **load-bearing** in the cryptography:

- The vault file-key derivation mixes the YubiKey `hmac-secret` output `R`:
  `KEK_key = HMAC-SHA256(R, mkfilekey3(user, pass))`, and the factotum blob is
  `encrypt3(vault, DK)` (AES-256-GCM) with the DK wrapped in the key slot
  (`doc/yubikey-2fa-operations.md` §9).
- At **UV**, `R` is the **WithUV CredRandom** — a *different value* than the touch-only
  output. Therefore a touch-only derive (no PIN) **cannot** unwrap a UV-enrolled vault.
- **Proof on hardware:** the standalone `t2uv` test enrolls a UV credential, derives twice
  *with* the PIN (must be deterministic) and once *without* (must differ). Both PASS on
  real hardware ⟹ the PIN is cryptographically required, not a UI gesture
  (`doc/yubikey-2fa-operations.md` §11; design in `doc/second-factor-auth.md`).

This is the PIV/CAC model: a hardware token whose use requires an on-device PIN, gating a
credential that never leaves the token.

## 3. Implementation (file-by-file evidence)

| Component | File | Role |
|-----------|------|------|
| Limbo 2FA interface | `module/twofa.m` | challenge/response + enroll API to `/mnt/2fa` |
| 2FA library | `appl/lib/twofa.b` | challenge/response, enroll helpers |
| Slot/KDF library | `appl/lib/secstore.b`, `appl/lib/twofaslot.b` | DK-wrapping slots; AES-256-GCM (SGCM2) vault, HKDF mixing of `R` |
| Login UI/flow | `appl/wm/logon.b` | PIN prompt when slots exist; key-derived unlock; recovery fallback |
| Host FIDO2 bridge | `emu/port/fido2bridge.*`, `emu/port/devtfa.c` | relays challenge→response to host libfido2; serves `#²`/`/dev/2fa` |
| Operator command | `2fa` (`appl/cmd/2fa.b`) | `status` / `enroll` / `disable` |

Lifecycle properties already met (`doc/yubikey-2fa-operations.md`):
- **Never-brick enroll:** every slot is built and verified to round-trip the DK *before*
  the live blob is replaced; blob swap rolls back on failure; recovery passphrase required
  up front (§3).
- **Controlled recovery:** a recovery slot (recovery-passphrase-wrapped DK) is the
  documented anti-lockout path (§7).
- **8-try hardware PIN lockout** on the key (§0, §10).

## 4. EPIC 1 closure status

Per EPIC 1 (`doc/security-epics.md`), the hardening stories are complete except a
deliberately-declined option:

| Story | AAL3 / hardening relevance | Status |
|-------|----------------------------|--------|
| (a) UV-required on enroll | the load-bearing AAL3 control | ✅ shipped, hardware-verified |
| (b) DK-encrypted save-back | no password-keyed blob is ever written for a 2FA account (closes the silent-downgrade gap) | ✅ shipped, hardware-verified |
| (c) Dual / backup key | ≥2 authenticators to avoid lockout | ✅ shipped, hardware-verified (`2fa addkey`, GUI; backup unlocks alone) |
| (e) Settings GUI panel | operability without a terminal | ✅ shipped |
| (g) End-to-end accreditation evidence | evidence completeness for an assessor | ✅ this artifact + `tests/twofaslot_test.b` (§4a) |
| (f) Passwordless mode | optional lower-friction posture | ⊘ **declined** — collapses both factors onto one device; an independent password factor is the conservative choice for a crown-jewel vault (still AAL3 either way) |

Residual threat-model exposure (carried, not a verifier gap): the host can observe a live
unlock (USB key, no pinpad), but the factor stays device-bound and at-rest/theft
resistance holds — `doc/second-factor-auth.md` §8.

## 4a. Test evidence (accreditation)

**Automated (CI, hardware-free)** — `tests/twofaslot_test.b`, run in the headless emu
(`o.emu -r$ROOT /tests/twofaslot_test.dis`); **6/6 PASS**:

| Test | Property asserted |
|------|-------------------|
| `NoSilentDowngrade` | a DK-encrypted vault is **not** decryptable with the password — the AAL3 at-rest invariant |
| `RecoverySlotRoundTrip` | the recovery slot recovers the exact data key |
| `WrongRecoveryFails` | a wrong recovery passphrase yields no key |
| `WriteslotsAdditive` | adding a slot never deletes one — the no-single-point-of-failure invariant |
| `DataKeyEnvelope` | the slot wrap/unwrap round-trips; a different key cannot unwrap the DK |
| `Mkkek2faDeterminism` | the slot KEK is deterministic and sensitive to both `R` and the rootkey |

**Manual (hardware, touch-required)** — repeatable checklist, all observed PASS on real
YubiKeys (procedures in `doc/yubikey-2fa-operations.md`):

1. **UV load-bearing** (`t2uv`): UV derive is deterministic *with* the PIN; a touch-only
   derive yields a *different* secret ⟹ the PIN is cryptographically required.
2. **Key-slot unlock**: a 2FA account logs in with key + (PIN) + touch.
3. **Backup-key independence**: after `2fa addkey`, the account unlocks with the **backup
   key alone** (primary removed).
4. **Recovery**: with the key absent the recovery passphrase unlocks; a wrong one does not.
5. **No downgrade on save-back**: a 2FA login reports `secstore save-back enabled
   (DK-encrypted)` and re-login round-trips.
6. **Single-key enroll guard**: enroll/addkey with two keys plugged fails with a clear
   "multiple security keys present" instruction.

## 5. Disposition

The **AAL3 verifier** — hardware authenticator + on-key user verification with a
cryptographically load-bearing PIN — is **implemented, hardware-verified, and
regression-tested**, with DK-encrypted save-back, dual/backup key, recovery, and a Settings
GUI all shipped. The one declined item (passwordless) is a deliberate defense-in-depth
choice, not a gap. **EPIC 1 (AAL3-harden the YubiKey login) is substantially complete and
this artifact is its accreditation evidence (g).** The broader program continues under the
umbrella epic [INFR-328] (audit-log AU-10 non-repudiation, dual-key-by-default policy,
formal threat-model doc). **Status: Met (AAL3 verifier); EPIC 1 closed.**
